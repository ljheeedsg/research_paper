# 数据质量实验方案：Data Quality Evaluation

## 1. 实验总目标

本组实验用于验证 **TruthRide** 在群智感知任务中是否能够持续获得**更高质量、更低误差、更可信的数据**。

你的研究场景是：每一轮平台发布任务，工人根据自己的轨迹、可执行任务集合和报价参与竞选。平台收到工人的数据后，需要判断这些数据是否可信，并在长期过程中尽量保留高质量工人、减少恶意工人或低质量工人的影响。

因此，数据质量实验不能只证明“平均质量高”，而要证明一条完整逻辑：

```text
TruthRide 获得更高数据质量
→ 上报数据误差更低
→ 信任验证能区分可信工人与恶意工人
→ 平台内部可信工人池逐渐扩大或稳定
→ 因此长期数据质量能够保持较高水平
```

这一组实验重点回答四个问题：

1. **整体效果**：TruthRide 的每轮数据质量和累计数据质量是否优于基准算法？
2. **误差效果**：TruthRide 的 RMSE 是否更低，即上报数据是否更接近可信基准？
3. **模块贡献**：Trust verification 和完整激励留存机制是否分别对数据质量有贡献？
4. **信任机制解释**：系统是否真的能让可信工人保持高信任、让恶意工人信任下降，并改善平台工人结构？

---

# 2. 数据集设置

本论文使用三个真实轨迹数据集：

1. **Beijing Taxi / T-Drive**
2. **Rome Taxi**
3. **Chicago Bike**

三个数据集不是随便堆上去，而是分别承担不同实验作用。

---

## 2.1 Beijing Taxi / T-Drive

### 数据集特点

北京出租车数据集轨迹密集，车辆数量较多，城市道路结构复杂，同一区域内不同车辆之间出现轨迹重叠的概率较高。

### 在本实验中的定位

北京数据集作为**主实验数据集**。

原因是你的信任验证机制依赖空间一致性，也就是需要在同一 grid / 同一任务区域内，有可信工人和未知工人同时或相近地经过。北京出租车轨迹密度较高，更容易形成这种交叉验证条件。

### 主要用于

1. 整体数据质量对比实验；
2. RMSE 对比实验；
3. 数据质量消融实验；
4. 信任验证有效性实验；
5. 三类工人状态变化趋势实验。

也就是说，**第一阶段所有数据质量主实验都先在北京数据集上做**。

---

## 2.2 Rome Taxi

### 数据集特点

罗马出租车数据集来自不同城市结构。相比北京，罗马城市道路更不规则，路网形态、轨迹分布和车辆移动模式都不同。

### 在本实验中的定位

罗马数据集作为**跨城市泛化验证数据集**。

它用来证明你的方法不是只在北京这种高密度轨迹场景下有效，而是在不同城市道路结构下仍然能够提高数据质量。

### 主要用于

后续扩展实验：

1. 不同数据集下最终累计数据质量对比；
2. 不同数据集下最终累计 RMSE 对比；
3. 不同数据集下 TruthRide 相对最优 baseline 的质量提升比例。

---

## 2.3 Chicago Bike

### 数据集特点

芝加哥自行车数据集与出租车数据不同。自行车速度较慢，轨迹更分散，参与更间歇，用户行为更不稳定。

### 在本实验中的定位

芝加哥数据集作为**动态参与和长期稳定性验证数据集**。

它非常适合验证：当工人参与不连续、轨迹稀疏、长期留存更困难时，TruthRide 是否仍然能够保持较好的数据质量。

### 主要用于

后续扩展实验：

1. 长期参与场景下的数据质量保持；
2. 激励与留存机制对数据质量的影响；
3. 不同数据集下最终累计数据质量和 RMSE 对比。

# 3. 对比算法设计

数据质量实验中有两类对比：

1. **整体算法对比**：用于实验一；
2. **模块消融对比**：用于实验二。

---

## 3.1 整体算法对比

实验一使用六个算法。

| 算法名称                        | 含义                               | 对比目的                        |
| --------------------------- | -------------------------------- | --------------------------- |
| **Random**                  | 随机选择可用工人                         | 作为下限基准，说明没有学习和筛选机制时数据质量较低   |
| **Explore-First / ε-First** | 前期随机探索，后期利用历史平均质量                | 对比简单探索-利用策略                 |
| **UCB-Greedy / CMAB**       | 基于 UCB 估计和边际收益/成本进行工人选择          | 验证基础招募机制对数据质量的提升            |
| **Trust-Aware**             | UCB-Greedy + 空间一致性信任验证           | 验证信任验证是否能过滤 false data，提高质量 |
| **Incentive-Aware**         | UCB-Greedy + Trust + 行为激励机制      | 验证激励机制是否能提高可信工人的参与度         |
| **TruthRide**               | UCB-Greedy + Trust + PGRD + LGSC | 完整方法，验证长期高质量数据获取能力          |

这六个算法形成递进关系：

```text
Random
→ Explore-First
→ UCB-Greedy
→ Trust-Aware
→ Incentive-Aware
→ TruthRide
```

这个对比链的意义是：

* Random 证明没有策略时效果较弱；
* Explore-First 证明简单探索利用机制有限；
* UCB-Greedy 证明基础招募可以提升质量；
* Trust-Aware 证明信任验证对质量提升有直接作用；
* Incentive-Aware 证明激励机制可以进一步促进可信工人参与；
* TruthRide 证明完整机制能够在长期过程中稳定获得高质量数据。

---

## 3.2 消融实验对比

实验二不建议继续画六条线。消融实验的目的不是和所有 baseline 比，而是验证你自己方法的模块贡献。

所以实验二确定使用三条线：

| 变体              | 含义                            | 验证目的               |
| --------------- | ----------------------------- | ------------------ |
| **UCB-Greedy**  | 只有基础招募机制                      | 无信任验证、无激励留存时的基础表现  |
| **Trust-Aware** | UCB-Greedy + 空间一致性验证          | 验证信任验证模块对数据质量的贡献   |
| **TruthRide**   | UCB-Greedy + Trust + 完整激励留存机制 | 验证完整机制是否进一步稳定高质量数据 |

这里**暂时不单独放 Membership-Aware 作为数据质量消融主线**。

原因是：

1. Membership-Aware / PGRD 的主要作用更偏向可信工人参与、会员转化和平台收益；
2. LGSC 的主要作用更偏向长期留存；
3. 如果在数据质量消融中加入太多中间版本，图会复杂，叙事会变散；
4. 数据质量消融的主问题是：**基础招募 → 加验证 → 加完整机制** 是否逐步提升质量。

所以实验二的消融逻辑确定为：

```text
基础招募
→ 加入信任验证
→ 加入完整激励与留存机制
```

# 4. 数据质量指标定义

本组实验确定使用三个主指标：

1. **Per-Round Average Quality**
2. **Cumulative Average Quality**
3. **Cumulative RMSE**

不使用 MAE 作为主指标。

---

## 4.1 为什么不使用 MAE

MAE 和 RMSE 都是误差指标，作用相近。
如果同时放 MAE 和 RMSE，容易显得指标重复。

RMSE 对较大误差更敏感，而你的研究重点之一就是识别 false data 和 malicious workers。恶意工人通常会产生较大的偏差，因此 RMSE 更适合作为主要误差指标。

所以确定：

```text
正文不放 MAE。
正文使用 RMSE 作为误差指标。
```

如果后续审稿人要求补充，可以把 MAE 放到附录或补充实验中，但当前实验方案不写 MAE。

---

## 4.2 Per-Round Average Quality

### 定义

每轮平均数据质量表示第 t 轮平台实际收到数据的平均质量。

对于第 t 轮中每个被覆盖任务 j，先计算该任务收到的所有工人上报数据的平均质量：

$q_j(t) = \text{average quality of submitted data for task j}$

然后对第 t 轮所有被覆盖任务求平均：

$$\text{AvgQuality}_t = \frac{1}{|C_t|} \sum_{j \in C_t} q_j(t)$$

其中：

* `C_t` 表示第 t 轮被至少一个工人执行或覆盖的任务集合；
* `q_j(t)` 表示任务 j 收到的数据平均质量；
* 没有任何工人执行的任务不参与本轮平均质量计算。

### 这个指标回答的问题

```text
当前这一轮，平台实际收到的数据质量高不高？
```

### 使用方式

这个指标放在实验一中，用来展示动态质量变化。

但是它会波动，因为每一轮：

* 任务位置不同；
* 可用工人不同；
* 工人轨迹覆盖不同；
* 被选择工人不同；
* 恶意工人参与情况不同。

所以正文建议使用：

```text
5-round moving average
```

或者：

```text
10-round moving average
```

如果轮数比较少，用 5-round moving average；如果轮数较多，用 10-round moving average。

---

## 4.3 Cumulative Average Quality

### 定义

累计平均数据质量表示从第 1 轮到第 t 轮，平台所有已覆盖任务的数据平均质量。

$$\text{CumAvgQuality}_t = \frac{\sum_{\tau=1}^{t} \sum_{j \in C_\tau} q_j(\tau)}{\sum_{\tau=1}^{t} |C_\tau|}$$

其中：

* `τ` 表示历史轮次；
* `C_τ` 表示第 τ 轮被覆盖的任务集合；
* `q_j(τ)` 表示第 τ 轮任务 j 的平均数据质量。

### 这个指标回答的问题

```text
从实验开始到当前轮，平台长期获得的数据质量整体如何？
```

### 使用方式

这是数据质量实验的**主指标**。

原因是：

1. 它比每轮质量更稳定；
2. 它能反映长期整体效果；
3. 它更适合论文正文；
4. 它能展示 TruthRide 是否持续保持高质量，而不是某几轮偶然表现好。

---

## 4.4 RMSE

RMSE 用于衡量工人上报数据与可信参考值之间的误差。

这里必须明确：**RMSE 不是每轮单个任务 RMSE 的求和。**

如果你把每个任务的 RMSE 直接相加，结果会受任务数量影响，不同轮次之间不可比，也不符合标准定义。

---

### 每轮 RMSE

第 t 轮的 RMSE 应该对所有可计算误差的数据点统一计算：

$$\text{RMSE}_t = \sqrt{\frac{1}{|D_t|} \sum_{(i,j) \in D_t} (x_{ij}(t) - x^*_j(t))^2}$$

其中：

* `x_ij(t)` 表示第 t 轮工人 i 对任务 j 上报的 `task_data`；
* `x*_j(t)` 表示任务 j 的可信参考值；
* `D_t` 表示第 t 轮所有可计算误差的 worker-task 数据点集合。

### 可信参考值怎么定义

参考值 `x*_j(t)` 建议使用：

```text
同一任务 / 同一 grid 中真实可信工人的 task_data 中位数
```

也就是说，如果一个任务或 grid 中有多个真实可信工人上报数据，就取这些可信工人数据的中位数作为参考值。

为什么用中位数？

因为中位数比平均值更不容易受异常值影响，更适合用作可信基准。

### 没有可信参考值怎么办

如果某个任务在当前轮没有真实可信工人提供参考数据，则这个任务不参与 RMSE 计算。

这需要在论文里写清楚：

```text
Tasks without trusted reference data are excluded from RMSE calculation.
```

这样做是合理的，因为没有 reference value 就不能定义误差。

---

### 累计 RMSE

正文建议使用 **Cumulative RMSE**。

累计 RMSE 不是把每轮 RMSE 加起来，而是把第 1 轮到第 t 轮所有可验证数据点放在一起统一计算：

$$\text{CumRMSE}_t = \sqrt{\frac{1}{|D_{1:t}|} \sum_{\tau=1}^{t} \sum_{(i,j) \in D_\tau} (x_{ij}(\tau) - x^*_j(\tau))^2}$$

其中：

* `D_{1:t}` 表示从第 1 轮到第 t 轮所有可计算参考误差的数据点集合。

### 为什么正文用 Cumulative RMSE

因为每轮 RMSE 也可能波动较大。
Cumulative RMSE 更稳定，更适合展示长期误差表现。

所以确定：

```text
实验一中正文放 Cumulative RMSE。
如果需要展示动态误差变化，可以附加 Per-Round RMSE，但不是主图。
```

# 5. 实验一：整体数据质量对比实验

2. TruthRide 的累计平均质量是否更高？
3. TruthRide 的累计 RMSE 是否更低？

---

## 5.2 数据集

实验一使用：

```text
Beijing Taxi / T-Drive
```

后续可以在 Rome Taxi 和 Chicago Bike 上重复，但第一阶段先使用 Beijing。

---

## 5.3 对比算法

实验一使用六条线：

1. Random
2. Explore-First / ε-First
3. UCB-Greedy / CMAB
4. Trust-Aware
5. Incentive-Aware
6. TruthRide

---

## 5.4 Fig. 1(a)：每轮平均数据质量

### 图设置

| 项目 | 内容                        |
| -- | ------------------------- |
| 横轴 | Round                     |
| 纵轴 | Per-Round Average Quality |
| 曲线 | 6 条算法曲线                   |
| 建议 | 5-round moving average    |

### 预期趋势

TruthRide 在多数轮次中保持最高或接近最高。
Incentive-Aware 和 Trust-Aware 次之。
UCB-Greedy 位于中间。
Random 和 Explore-First 较低，并且波动较大。

### 为什么会出现这个趋势

Random 没有任何学习和筛选机制，因此会随机选择低质量工人、普通工人甚至恶意工人，平均质量通常较低。

Explore-First 前期通过随机探索收集信息，后期利用历史质量选择工人。但如果前期探索不充分，或者探索阶段选中了较多低质量/恶意工人，后期可能会锁定次优工人，因此质量提升有限。

UCB-Greedy 能通过历史质量和 UCB 探索项逐步选择更优工人，所以会优于 Random 和 Explore-First。但是它没有显式验证 false data，因此仍可能被虚假上报工人影响。

Trust-Aware 引入空间一致性验证，通过可信工人的参考数据降低恶意工人的 trust，使恶意工人后续更少被选择，因此数据质量提高。

Incentive-Aware 通过激励机制提高可信工人的参与意愿，使平台更容易获得高质量数据。

TruthRide 在 Trust 和激励基础上进一步加入长期留存机制，使可信工人能够持续参与，因此中后期数据质量最稳定。

### 图设置

| 项目 | 内容                         |
| -- | -------------------------- |
| 横轴 | Round                      |
| 纵轴 | Cumulative Average Quality |
| 曲线 | 6 条算法曲线                    |

### 预期趋势

整体预期为：

```text
TruthRide > Incentive-Aware > Trust-Aware > UCB-Greedy > Explore-First / Random
```

如果实际实验中 Incentive-Aware 和 Trust-Aware 接近，也是可以解释的。因为激励机制主要作用于参与和留存，它对数据质量的提升可能更多体现在中后期，而不是每一轮都明显高于 Trust-Aware。

### 为什么会出现这个趋势

累计平均质量消除了单轮任务随机性的影响，更能反映长期整体效果。

TruthRide 之所以累计质量最高，是因为它同时解决了三个问题：

1. UCB-Greedy 负责选择历史表现较好的工人；
2. Trust verification 负责过滤虚假数据和低信任工人；
3. PGRD/LGSC 负责增强可信工人的长期参与。

因此，随着轮次增加，TruthRide 的高质量数据来源更加稳定，累计平均质量应逐渐领先。

### 图设置

### 为什么 RMSE 图不一定画六条线

RMSE 图的目的主要是证明 false data 误差控制能力。画四条线已经能表达清楚：

```text
无策略下限
→ 基础招募
→ 加入信任验证
→ 完整方法
```

如果画六条线，图会比较乱，且 Incentive-Aware 与 TruthRide 的 RMSE 可能非常接近，不一定提升表达效果。

### 预期趋势

TruthRide 的 Cumulative RMSE 最低。
Trust-Aware 次低。
UCB-Greedy 和 Random 较高。

### 为什么会出现这个趋势

Random 会随机选择恶意或低质量工人，因此上报数据更容易偏离可信参考值。

UCB-Greedy 虽然能基于历史质量逐渐偏向高质量工人，但它没有显式空间一致性验证，因此无法稳定识别虚假上报。

Trust-Aware 能用真实可信工人的数据作为参考，发现与参考值偏差较大的工人，并降低其 trust，因此 RMSE 下降。

TruthRide 进一步通过长期激励留住可信工人，使参考数据来源和高质量数据来源更稳定，因此累计 RMSE 最低。

### 预期趋势

```text
TruthRide > Trust-Aware > UCB-Greedy
```

### 为什么会出现这个趋势

UCB-Greedy 只能基于历史质量估计和边际收益选择工人。当存在恶意工人时，它可能无法准确区分真实高质量数据和伪造数据。

Trust-Aware 加入验证任务后，能够利用空间一致性比较 unknown worker 和 trusted worker 的数据差异，从而降低恶意工人的 trust，并减少其后续参与。因此，Trust-Aware 的累计平均质量应高于 UCB-Greedy。

TruthRide 在 Trust-Aware 基础上加入 PGRD 和 LGSC。PGRD 提高可信工人参与意愿，LGSC 降低高质量工人流失，因此中后期可信工人池更稳定，累计质量进一步提升。

## 7.3 统计口径

这一点必须写清楚：

```text
按真实初始身份 init_category 分组，
统计每一轮平台当前维护的 worker["trust"]。
```

也就是说：

* 分组依据：真实标签 `init_category`
* 统计值：当前轮的 `worker["trust"]`
* 不是按平台当前 `category` 分组

原因是：

你真正要证明的是：

所以必须按真实标签分组。

### 定义

$$\text{AvgTrust}_{\text{trusted}}(t) = \text{average current trust of workers whose init\_category == "trusted"}$$

真实可信工人的数据与可信参考值较接近。在验证任务中，它们不会被持续惩罚。如果其数据误差低于设定阈值，trust 还会进一步提高或保持高值。

| -- | ------------------------------------------ |
| 横轴 | Round                                      |
| 纵轴 | Average Trust                              |

## 7.7 Fig. 3(d)：Malicious Detection F1-score

### 这个指标是什么

Malicious Detection F1-score 用于衡量系统识别恶意工人的准确性。

比较的是：

```text
真实标签：init_category == malicious
系统标签：category == malicious
```

### 定义

```text
TP = 真实是 malicious，系统也识别为 malicious
FP = 真实不是 malicious，但系统误判为 malicious
FN = 真实是 malicious，但系统没有识别为 malicious
```

然后：

$$\text{Precision} = \frac{TP}{TP + FP}$$

$$\text{Recall} = \frac{TP}{TP + FN}$$

$$F1 = \frac{2 \times \text{Precision} \times \text{Recall}}{\text{Precision} + \text{Recall}}$$

### 直观解释

Precision 高，说明系统识别出来的 malicious 工人大多数确实是恶意工人。

Recall 高，说明真实恶意工人大多数都被系统找出来了。

F1-score 高，说明系统识别恶意工人既准又全。

### 图设置

| 项目 | 内容                           |
| -- | ---------------------------- |
| 横轴 | Round                        |
| 纵轴 | Malicious Detection F1-score |
| 曲线 | Trust-Aware、TruthRide        |

### 预期趋势

F1-score 前期较低，中期逐渐上升，后期趋于稳定。

### 为什么会出现这个趋势

前期系统验证信息不足，很多恶意工人还没有被充分观察，因此识别准确率较低。随着验证任务积累，恶意工人的 false data 被逐渐暴露，系统能更准确地将其识别为 malicious。
