# 群智感知双阶段工人招募与信任度验证算法（三阶段架构）

## 概述

本算法将整个流程划分为三个独立阶段，便于逐步实现与调试。  
- **第一阶段：数据准备**：生成工人可选项、任务权重，初始化工人信任度与任务数据，建立任务网格映射。  
- **第二阶段：初始化（学习阶段）**：招募所有工人，学习初始质量，建立工人档案和任务覆盖状态，初始化工人分类集合。  
- **第三阶段：贪心轮次（多轮迭代）**：按时间轮次招募工人，同时进行信任度验证（基于空间一致性），动态更新信任度，淘汰恶意工人。  

**核心思想**：每轮开始时，根据当前轮次的可用工人（由工人ID中的小时决定）和可用任务（由任务开始时间决定），进行招募。验证任务基于**本轮可用工人的轨迹分布**生成，优先选择有可信工人经过的网格，并按未知工人出现次数排序选取前 M 个，确保基准数据存在且能顺带完成。验证任务由所有覆盖该任务的工人（而不只是招募的K个工人）执行，因此每轮可以批量验证大量工人，快速识别可信/恶意工人。虽然这会产生额外成本（未知工人执行验证任务需支付报酬），但通过控制验证任务数量M可调节。

# 第一阶段：数据准备

## 1.1 目标
- 根据原始轨迹和任务数据，为每个工人生成可覆盖的任务列表（含质量、报酬、时间交集）。
- 为每个工人设置初始信任度与类别（可信/未知）。
- 为每个工人覆盖的任务随机生成模拟上报数据（用于后续验证）。
- 建立任务到网格的映射，为空间验证提供基础。
- 生成任务权重（所需工人数）。

## 1.2 输入
- `step1_worker_segments.json`：按区域分组的工人轨迹段，每个段包含 `vehicle_id`、`region_id`、`start_time`、`end_time`、`cost`、`is_trusted`。
- `step1_task_segments.json`：按区域分组的任务窗口，每个任务包含 `task_id`、`region_id`、`start_time`、`end_time`、`required_workers`。

## 1.3 处理步骤

### 1.3.1 解析数据
- 将工人轨迹段按 `worker_id` 聚合，每个工人得到一个轨迹段列表（每个段包含区域、时间窗口、报价、可信标签）。
- 将所有任务平铺为一个列表，每个任务包含区域、时间窗口、所需工人数。

### 1.3.2 计算工人可选项
对每个工人：
- 初始化空列表 `covered_tasks`。
- 遍历所有任务，判断工人是否能覆盖该任务：
  - **覆盖条件**：工人与任务区域相同，且时间窗口有交集（即 `worker.start < task.end` 且 `worker.end > task.start`）。
  - 若满足条件：
    - **交集时间**：
      \[
      \text{start} = \max(\text{worker.start}, \text{task.start})
      \]
      \[
      \text{end} = \min(\text{worker.end}, \text{task.end})
      \]
    - **报酬**：`task_price = worker.cost`（工人的固定报价）。
    - **质量**：在 `[0,1]` 内均匀随机生成（不依赖可信度）。
    - **任务数据**：随机生成 `task_data ∈ [0,1]`，模拟工人上报数据。
    - 将任务信息（`task_id`, `quality`, `task_price`, `start_time`, `end_time`, `task_data`）加入 `covered_tasks`。
- 计算工人总成本：`total_cost = len(covered_tasks) × worker.cost`。
- 根据工人的 `is_trusted` 初始化信任度与类别：
  - 若 `is_trusted == True`：`trust = 1.0`，`category = "trusted"`。
  - 若 `is_trusted == False`：`trust = 0.5`，`category = "unknown"`。
- 记录工人的 `worker_id`、`is_trusted`、`trust`、`category`、`total_cost`、`covered_tasks`。

### 1.3.3 生成任务权重
- 任务权重 = 任务原始数据中的 `required_workers`。

### 1.3.4 生成任务网格映射
- 遍历 `step1_task_segments.json` 中的所有任务，提取 `task_id` 和 `region_id`。
- 设置 `grid_id = region_id`（直接用区域 ID 作为网格 ID）。
- 输出为 JSON 列表。

# 第二阶段：初始化（学习阶段）

## 2.1 目标
- 建立工人的初始 CMAB 质量档案。
- 初始化任务覆盖状态。
- 根据第一阶段的数据初始化工人分类集合 `Uc`、`Uu`、`Um`。
- 初始化全局变量（总学习次数等）。

## 2.2 输入
- `step5_worker_option_set.json`
- `step5_task_weight_list.json`

## 2.3 处理步骤

1. **加载数据**：读取工人选项和任务权重。
2. **初始化工人 CMAB 档案**：
   - 学习次数 `n_i = len(covered_tasks)`。
   - 平均感知质量 `avg_quality_i = (∑ quality) / n_i`（若无覆盖任务则为 0）。
   - 判断次数（用于统计）`judge_count = 1`。
3. **初始化工人分类集合**：
   - 根据每个工人的 `category` 字段，将其 ID 分别加入 `Uc`（可信）、`Uu`（未知）、`Um`（恶意，初始为空）。
4. **初始化任务覆盖**：所有任务的覆盖计数置为 0。
5. **计算总学习次数**：`total_learned_counts = ∑ n_i`。

## 2.4 输出（内部状态）
- 工人档案（扩展了 CMAB 字段）
- 任务覆盖计数字典
- 工人分类集合 `Uc`, `Uu`, `Um`
- 全局变量 `total_learned_counts`

# 第三阶段：贪心轮次（多轮迭代，t=1 至 R）

## 3.1 目标
- 按时间轮次招募工人，最大化任务覆盖。
- 每轮进行信任度验证（基于当前轮工人轨迹分布），动态更新信任度，逐步淘汰恶意工人。

## 3.2 输入
- 第二阶段输出的工人档案、任务覆盖状态、工人分类集合、全局变量。
- 参数：总预算 `B`，每轮招募人数 `K`，轮数 `R=6`，信任度更新步长 `η`，可信阈值 `θ_high`，恶意阈值 `θ_low`，每轮验证任务数 `M`。
- 任务网格映射 `step5_tasks_grid_num.json`。

## 3.3 每轮处理步骤（循环 t=1..R）

### 3.3.1 确定当前轮可用工人与任务
- **可用工人**：`worker_id` 中的小时字段等于当前轮次 `t-1` 的工人（因为第1轮对应小时0，第2轮对应小时1，依此类推）。
- **可用任务**：任务开始时间 `start_time` 所在轮次等于 `t-1` 的任务（即 `start_time // 3600 == t-1`）。

### 3.3.2 终止条件检查
- 若剩余预算 < 当前轮可用工人中的最小成本，终止。
- 若所有可用任务已完成（覆盖计数达到所需工人数），终止。
- 若达到最大轮数，终止。

### 3.3.3 生成验证任务（基于当前轮可用工人的轨迹）
- **网格统计**：  
  对每个当前轮可用的工人，根据其覆盖的任务（每个任务对应一个网格），统计每个网格中 `Uc` 工人出现次数和 `Uu` 工人出现次数。  
  （注意：只统计**本轮可用工人**，因为只有这些工人会实际执行任务。）
- **筛选**：仅保留 `Uc` 出现次数 > 0 的网格（即有可信工人经过的网格）。若没有这样的网格，则本轮不生成验证任务。
- **排序**：将筛选后的网格按 `Uu` 出现次数降序排列，取前 `M` 个网格。
- **任务选取**：对每个选中网格，从该网格的**可用任务列表**中**随机选取一个任务**作为验证任务（确保该任务属于当前轮次）。
- 生成验证任务列表 `validation_tasks`（实际数量可能少于 M，如果可信网格不足）。

**说明**：只选择有可信工人经过的网格，确保每个验证任务都有可信工人作为基准。可信工人可以顺路完成验证任务，其成本已包含在招募中，因此不增加额外开销；未知工人执行验证任务会产生成本，但通过控制 M 可调节。若可信网格不足，验证任务数量会相应减少，避免为无基准的任务支付成本。

### 3.3.4 发布任务
- 业务任务：当前轮可用的所有原始任务（权重为 `required_workers`）。
- 验证任务：`validation_tasks`（权重设为 0，不参与业务增益计算，但工人执行会消耗成本）。

**执行范围**：验证任务由所有**本轮可用的工人**（即其轨迹覆盖当前轮次的工人）执行，因为这些工人在本轮时间窗口内可以顺路完成验证任务。因此，每轮可能有大量工人参与验证，信任度更新可同时影响许多工人，实现快速批量识别可信/恶意工人。每个工人的 `task_data` 是预先随机生成并固定，因此对同一验证任务，同一工人每次上报的数据相同。

**成本说明**：工人执行验证任务同样会获得报酬（`task_price` 已计入其总成本），因此每轮验证任务会额外增加平台支出。这是快速验证的代价，可通过参数 `M` 控制验证任务数量来调节成本与验证效率的平衡。

### 3.3.5 招募工人（CMAB 贪心选择）
- 候选工人 = `Uc ∪ Uu`（排除 `Um`），且仅限当前轮可用的工人。
- 使用 UCB 贪心策略，每轮选择 `K` 个工人：
  1. 对每个候选工人，计算 UCB 质量：
     \[
     \hat{q}_i(t) = \bar{q}_i(t-1) + \sqrt{\frac{(K+1)\ln(T_{\text{learn}})}{n_i(t-1)}}
     \]
  2. 计算边际增益：
     \[
     \text{gain}_i = \sum_{j \in \mathcal{M}_i^{\text{avail}} \cap \mathcal{M}_i^{\text{uncov}}} w_j \cdot \hat{q}_i(t)
     \]
     - \(\mathcal{M}_i^{\text{avail}}\)：工人可覆盖且属于当前轮次的任务。
     - \(\mathcal{M}_i^{\text{uncov}}\)：这些任务中尚未完成的任务（覆盖计数 < \(w_j\)）。
  3. 性价比 = `gain / total_cost`。
  4. 选择性价比最高的工人。
- 支付成本，更新 `total_cost` 和 `remaining_budget`。
- 更新任务覆盖计数：对选中的工人，将其覆盖的未完成任务的计数加1（但不超过 `required_workers`）。
- 更新工人档案：
  - 新增学习次数 = 工人本轮实际贡献的任务数（即 `|ℳ_i^{avail} ∩ ℳ_i^{uncov}|`）。
  - `n_i` 增加。
  - `avg_quality` 按加权平均更新。
  - `total_learned_counts` 增加。

### 3.3.6 信任度更新（基于验证任务）
- 对于每个验证任务 `v`：
  1. 收集所有完成该任务的 `Uc` 工人的 `task_data`，取中位数作为基准 `base`。由于生成时已保证有 Uc 工人，此步骤通常有数据。若因特殊情况（如可信工人未完成该任务）导致无数据，则跳过该任务。
  2. 对于每个完成该任务的 `Uu` 工人 `i`（**仅限本轮可用的工人**，即其轨迹覆盖当前轮次的工人），获取其上报数据 `data`：
     - 计算误差：
       \[
       \text{error} = \frac{|data - base|}{base} \quad (\text{若 } base = 0 \text{ 则 } error = |data - base|)
       \]
     - 更新信任度：
       \[
       \text{trust}_i \leftarrow \text{trust}_i + \eta (1 - 2\cdot\text{error})
       \]
       \[
       \text{trust}_i \leftarrow \max(0, \min(1, \text{trust}_i))
       \]
     - 若 `trust_i ≥ θ_high`，将工人从 `Uu` 移入 `Uc`，更新 `category`。
     - 若 `trust_i ≤ θ_low`，将工人从 `Uu` 移入 `Um`，更新 `category`。

**说明**：每个验证任务都有可信工人作为基准，因此信任更新可正常进行。

### 3.3.7 输出本轮统计
- 打印：轮次、招募人数、总花费、剩余预算、当前轮已完成任务数、`|Uc|`、`|Uu|`、`|Um|`。

## 3.4 输出

最终输出 JSON 文件，包含：
- `total_rounds`：实际贪心轮数
- `total_cost`：总花费（仅贪心阶段的成本）
- `remaining_budget`：剩余预算
- `selected_workers`：贪心阶段招募的工人ID列表
- `init_select`：初始轮涉及的工人数（所有工人）
- `later_select`：贪心阶段招募的工人总次数
- `covered_task_count`：最终覆盖的任务数
- `trusted_count`：最终可信工人数
- `malicious_count`：最终恶意工人数
- `unknown_count`：最终未知工人数
- `trusted_workers_list`：最终可信工人 ID 列表

|------|------|--------|----------|

## 总结

本架构通过将验证任务生成限定在有可信工人经过的网格，并优先选取未知工人出现最多的网格，确保了基准数据的可靠性，同时利用可信工人顺路完成验证任务，降低了额外成本。验证任务由所有覆盖该任务的工人（而非仅招募的K个工人）执行，实现了低成本快速批量信任更新。通过调整参数 M 可以平衡验证效率与预算消耗。该设计使得平台能在有限预算下，高效识别并淘汰恶意工人，提升数据质量。后续可在此基础上继续添加会员激励模块（PGRD、LGSC），只需在第三阶段的循环中嵌入相应函数即可。
